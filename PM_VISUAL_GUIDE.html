Workers

D1 Database
The Permanent Record
All 60 feedback records and their AI analyses live here. Two tables: raw feedback (immutable) and analysis (replaceable). Complex queries supported via SQL.
The Analogy Two filing cabinets: one for original documents (never altered), one for analyst reports (can be rewritten if you hire a better analyst).
SQLite-compatible 2 tables, 40 columns Serverless

Workers AI
The Analyst
Runs Llama 3.1-8B on Cloudflare's own GPUs. Reads raw feedback and returns 22 structured fields: sentiment, urgency, competitor, PII, themes, and more. No external API.
The Analogy An in-house analyst who reads every piece of feedback and hands you back a completed, structured assessment form — in 2 seconds flat.
Llama 3.1-8B 22 output fields ~2s per record

Workflows
The Reliable Orchestrator
A 6-step durable pipeline that processes each new feedback record. If any step fails, it retries automatically. State is saved between steps — no data loss ever.
The Analogy A checklist that saves itself after each tick — if the power goes out halfway through, you pick up exactly where you left off.
6 steps Auto-retry Durable state
📌
KV Store
The Speed Cache
Stores pre-computed results for 5 minutes. Dashboard stats, segment data, and the AI digest are cached here. Turns 200ms D1 queries into 5ms KV reads.
The Analogy A sticky note on the fridge: instead of cooking dinner every time someone is hungry, the note says "pizza, ready at 7pm, expires in 5 minutes."
~5ms reads 5min TTL (stats) 1hr TTL (digest)
📦
R2 Storage
The Archive
Stores full raw payloads (Zoom transcripts, lengthy emails, sales call notes) long-term. D1 just stores a reference pointer. S3-compatible, but with zero egress fees.
The Analogy The archive room: the filing cabinet holds a card saying "see box 47 in archive room." The 500-page document lives in the archive, not in the cabinet.
$0 egress fees S3-compatible Unlimited size

Access / Zero Trust
The Identity Gate
Blocks all unauthenticated traffic before it reaches the Worker. Integrates with Google/GitHub SSO. Optional WARP device posture check for sensitive content.
The Analogy The security guard who checks your badge at the building entrance — every time, regardless of whether you were here yesterday.
Free ≤50 users SSO + WARP No code changes

PM Insight Cards
💡 "Why not just use AWS? Everyone uses AWS."
AWS is excellent — but it requires significant DevOps investment. For an internal PM tool that needs to ship fast, Cloudflare eliminates the need for load balancers, auto-scaling groups, RDS setup, and IAM configuration. The full stack (database, cache, AI, auth, runtime) is integrated and pre-configured.
💡 "Is Cloudflare just a CDN company?"
That's their origin (2010), but today the developer platform (Workers, D1, KV, R2, AI) is their fastest-growing segment. They're competing directly with AWS Lambda, DynamoDB, S3, and SageMaker. The CDN/security products still generate most revenue, but the developer platform is the strategic bet.
💡 "What's the biggest risk of betting on Cloudflare?"
Platform coupling. Workers code is written to Cloudflare's APIs — not portable to Lambda. D1 SQL is exportable (it's SQLite), but Workflows and Workers AI would need to be rewritten. For a product at early stage, speed beats portability. At scale, the switching cost grows. This is a deliberate PM trade-off to make consciously.
💡 "How does pricing scale as we grow?"
Workers: $0.50 per million requests. At 1,000 requests/day (internal tool), that's essentially free. D1: $0.001 per 1,000 reads. KV: $0.50 per million reads. The cost profile is extremely favorable for low-to-medium traffic internal tools and scales linearly, not exponentially.
Interview Prep

Questions & Answers

Click any question to reveal the PM-framed answer. These cover what you're likely to be asked — and what you should ask.

Why did you use Cloudflare Workers instead of a traditional server? ▼

Three reasons. First, time-to-ship: the entire infrastructure stack (database, cache, AI, orchestration, auth) is pre-integrated — no weeks of DevOps setup. Second, cost alignment: pay-per-request matches an internal tool's variable usage pattern far better than paying for a server running 24/7. Third, global performance: Workers runs at 300+ edge locations automatically — the PM in Singapore gets the same response time as the one in London. The trade-off is platform coupling: Workers code isn't portable to AWS Lambda without rewriting. For a product at this stage, speed of execution wins.

How does your AI analysis work? What makes it reliable? ▼

Workers AI runs Llama 3.1-8B on Cloudflare's own GPU infrastructure — no external dependency on OpenAI or Anthropic. Each feedback record is processed with structured extraction: one carefully engineered prompt goes in, one JSON object with 22 typed fields comes out. This isn't a chatbot — it's a classifier. Reliability comes from two mechanisms: (1) the parser validates every field and applies safe defaults if the AI returns unexpected output — bad AI doesn't crash the system, it degrades gracefully; (2) each analysis includes a confidence score, and low-confidence records can be flagged for manual review. At 96% average confidence across the seed data, the system is production-quality.

What happens if Cloudflare has an outage? ▼

Cloudflare's network operates at 99.99%+ uptime — higher than a single-region AWS deployment. But the more precise answer is about individual service failure modes. If Workers AI is briefly unavailable, Workflows retries the analysis step automatically (2 retry attempts with backoff). If that also fails, the feedback is stored with status "failed" and can be re-queued. If D1 is slow, the Worker can still serve cached stats from KV. No single point of failure takes down the entire product. The fallback path in the Worker even runs AI synchronously if the Workflow system itself is unavailable — three layers of resilience.

How does the PII handling work — and why does it matter for PMs? ▼

Workers AI detects PII (emails, IPs, API keys, names) during analysis and flags the record. The parser immediately generates a redacted_summary where PII is replaced with tokens like [REDACTED_EMAIL]. This happens in the analysis pipeline — not at display time. This means PII never appears in any API response, regardless of which dashboard view or filter you use. Critically, this is enforced server-side: even if a developer inspects network traffic in browser DevTools, they only see redacted content. For PMs, this means you can share screenshots of the dashboard in Slack or presentations without accidentally leaking customer contact information.

How would you scale this to 100,000 feedback records? ▼

Five changes: (1) Move theme aggregation from JavaScript (loads all records into memory) to SQL GROUP BY — this is the most urgent bottleneck. (2) Extend KV caching to cover more expensive queries (themes table, competitor table). (3) Add composite D1 indexes for the most common filter combinations (e.g., urgency + sentiment + product_category). (4) Add read-only D1 replicas for dashboard queries, keeping the primary for writes. (5) Consider archiving records older than 90 days to R2 in JSON format and only keeping recent records in the active D1 for fast dashboard queries. The architecture doesn't need to change fundamentally — these are all optimization layers on the existing design.

Why did you choose not to implement geospatial features? ▼

Evaluated it and deliberately decided against it. There's no structured geographic field in the schema — "eu-west" appears in the raw text of 2-3 records out of 60, which isn't a queryable attribute. Building a geographic view on substring matching (LIKE '%eu%') would be decorative rather than meaningful, and potentially misleading. The segment-based breakdown (Enterprise / Mid-Market / Emerging) delivers more actionable PM intelligence because it's backed by structured, AI-classified data. A geospatial view is on the V2 roadmap — but it requires adding a location field at the ingestion layer first, not retrofitting it to existing data.

What would V2 of SignalDesk look like? ▼

V2 closes the loop: V1 surfaces signal; V2 enables action. Specifically: (1) Two-way Slack integration — PM can respond to a feedback card from Slack and the response is logged in SignalDesk. (2) Jira/Linear integration — one click to convert a high-urgency record into a tracked issue. (3) Semantic deduplication using Cloudflare Vectorize (vector database) — identify that "Workers cold start" and "slow startup times" are the same issue even with different wording. (4) Trend alerting — background Workflow that runs nightly and sends a Slack summary if any theme volume increased >40% week-over-week. (5) Multi-tenant support — separate the data layer per customer so multiple companies can use SignalDesk as a SaaS product.
